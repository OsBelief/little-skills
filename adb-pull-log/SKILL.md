# ADB 日志拉取工具

这个 skill 帮助你通过 ADB 从 Android TV 设备拉取日志文件到本地计算机。

## 何时使用

- 用户需要从 Android TV 设备拉取日志文件
- 需要下载 logcat、kernel log、crash log 等日志文件
- 需要获取设备日志进行分析或调试
- 用户提到"拉取日志"、"下载日志"、"adb pull 日志"、"获取 logcat"等类似表述
- 需要批量拉取 /data/log/ 目录下的所有日志文件

### 命令行调用

**通过斜杠命令快速调用：**
```
/adb-pull-log [设备地址] [文件名]
```

**示例：**
```
# 使用默认参数（拉取默认设备的 logcat_full.log）
/adb-pull-log

# 指定设备地址
/adb-pull-log 172.33.50.2:5353

# 指定设备和文件名
/adb-pull-log 172.33.50.2:5353 logcat_full.log

# 只指定文件名（使用默认设备）
/adb-pull-log "" kernel.log

# 拉取所有日志文件
/adb-pull-log 172.33.50.2:5353 "*.log"
```

## 默认参数

- **设备地址**：`172.33.50.2:5353`
- **源目录**：`/data/log/`
- **默认文件**：`logcat_full.log`
- **目标目录**：`~/Desktop/`

## 工作流程

### 步骤 1：解析参数

**从命令行或用户消息中提取参数：**

1. **设备地址**（格式：`IP:PORT` 或 `IP`）
   - 如果用户提供了设备地址，使用用户提供的值
   - 如果用户没有提供，使用默认地址 `172.33.50.2:5353`

2. **源文件名**
   - 如果用户提供了文件名（如 `logcat_full.log`、`kernel.log`），使用用户提供的值
   - 如果用户没有提供，使用默认文件 `logcat_full.log`
   - 支持通配符，如 `*.log` 表示拉取所有日志文件

3. **目标文件名**
   - 默认使用源文件名作为目标文件名
   - 如果用户提供了不同的目标文件名，使用用户指定的名称

### 步骤 2：检查设备连接和 root 权限

在拉取日志之前，需要确保：
1. 设备已通过 ADB 连接
2. 设备已获取 root 权限（某些日志需要 root 权限才能读取）

**检查连接状态：**
```bash
adb devices | grep <设备地址>
```

**检查 root 权限：**
```bash
adb -s <设备地址> shell whoami
```

**如果设备未连接或无 root 权限：**
1. 调用 `/adb-connect <设备地址>` skill
2. 等待 adb-connect 执行完成
3. **重新检查**连接状态和权限：
   ```bash
   # 重新检查连接
   adb devices | grep <设备地址>

   # 重新检查权限
   adb -s <设备地址> shell whoami
   ```
4. **如果验证成功**（设备已连接且权限为 root）：
   - 继续执行后续步骤（步骤 3）
   - 在输出中说明已成功通过 adb-connect 获取权限
5. **如果验证失败**：
   - 向用户报告错误
   - 说明哪个步骤失败（连接失败或权限获取失败）
   - 停止执行后续步骤

### 步骤 3：检查源文件是否存在

在拉取之前，验证源文件是否存在：

```bash
adb -s <设备地址> shell ls -lh /data/log/<文件名>
```

**处理结果：**
- 如果文件存在，继续执行拉取操作
- 如果文件不存在，显示错误信息并询问用户是否要列出可用的日志文件
- 可选择的操作：显示 `/data/log/` 目录下的文件列表

### 步骤 4：检查本地目标文件

**如果目标文件已存在于 Desktop：**
- 自动重命名目标文件，添加时间戳
- 格式：`<文件名>_<年月日>_<时分秒>.log`
- 示例：`logcat_full_20250317_143020.log`

**检查命令：**
```bash
ls -lh ~/Desktop/<目标文件名> 2>/dev/null
```

**生成新的文件名：**
```bash
# 使用当前时间戳
TIMESTAMP=$(date +"%Y%m%d_%H%M%S")
NEW_FILENAME="${filename%.log}_${TIMESTAMP}.log"
```

### 步骤 5：拉取日志文件

**执行 adb pull 命令：**

```bash
adb -s <设备地址> pull /data/log/<源文件名> ~/Desktop/<目标文件名>
```

**拉取单个文件示例：**
```bash
adb -s 172.33.50.2:5353 pull /data/log/logcat_full.log ~/Desktop/logcat_full.log
```

**拉取多个文件（通配符）：**
如果用户请求拉取所有日志文件（如 `*.log`），需要：
1. 先获取文件列表：`adb -s <设备地址> shell ls /data/log/*.log`
2. 逐个拉取每个文件
3. 为每个文件生成唯一的文件名（带时间戳）

### 步骤 6：显示文件信息

**拉取成功后，显示以下信息：**

1. **目标文件信息**：
   - 本地保存路径
   - 本地文件大小
   - 文件行数
   - 是否被重命名（如果原文件已存在）

## 错误处理原则

**关键改进：自动重试机制**

遇到以下错误时，会自动调用 adb-connect skill 并重试：

1. **设备未连接**
   - 调用 `/adb-connect <设备地址>` skill
   - 等待连接完成
   - **重新验证**连接状态
   - 如果连接成功 → 继续执行后续步骤
   - 如果连接失败 → 停止执行，向用户报告错误

2. **无 root 权限**
   - 调用 `/adb-connect <设备地址>` skill
   - 等待权限获取完成
   - **重新验证**权限状态（`whoami` 返回 `root`）
   - 如果权限获取成功 → 继续执行后续步骤
   - 如果权限获取失败 → 停止执行，向用户报告错误

3. **源文件不存在**：显示错误信息并列出可用的日志文件
4. **权限不足**：明确告知用户需要 root 权限，提供解决方案
5. **本地文件系统错误**：检查 Desktop 目录是否存在，是否有写权限
6. **传输中断**：显示错误信息，建议用户重试
7. 不要尝试自动修复多次失败的操作（最多重试 1 次）
8. 保留所有步骤的执行状态，让用户了解进度

## 常见日志文件说明

Android TV 设备常见的日志文件包括：

- **logcat_full.log** - 完整的 logcat 日志，包含所有级别的日志信息
- **kernel.log** - 内核日志，记录系统底层运行信息
- **crash.log** - 崩溃日志，记录应用和系统崩溃信息
- **anr.log** - ANR（Application Not Responding）日志
- **boot.log** - 启动日志，记录系统启动过程

## 注意事项

1. **网络连接**：设备和计算机需要在同一网络
2. **存储空间**：确保 Desktop 有足够的存储空间
3. **文件大小**：日志文件可能很大（数百 MB），拉取可能需要较长时间
4. **权限要求**：某些日志文件需要 root 权限才能读取
5. **文件覆盖**：同名文件会自动重命名，不会覆盖
6. **批量拉取**：拉取多个文件时，每个文件都会添加时间戳避免重名
7. **路径正确性**：默认源路径是 `/data/log/`，某些设备可能使用不同的路径
8. **自动重试机制**：
   - 当检测到设备未连接或无 root 权限时，会自动调用 adb-connect skill
   - adb-connect 会尝试连接设备、配置调试参数、获取 root 权限
   - 获取成功后会自动重试拉取操作，无需手动干预
   - 最多自动重试 1 次，如果仍然失败会停止执行并报告错误

## 依赖工具

- **adb**：Android Debug Bridge，必须已安装并在系统 PATH 中
- **adb-connect skill**：用于设备连接和权限获取
- **标准 Unix 工具**：ls、wc、head、date 等

- 自动处理文件名冲突，不需要用户干预
- 批量操作时显示每个文件的处理进度
- 使用表情符号和格式化增强可读性
